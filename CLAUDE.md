# CLAUDE.md

Guidance for AI assistants (and humans) working in this repository.

## Project overview

Chrome extension (Manifest V3) that color-codes the Life Cycle Dates table on
https://access.redhat.com/support/policy/updates/openshift by deadline proximity.

## Documentation rules

- **README.md / README.jp.md are user-facing only** (what it does, install,
  settings, privacy, disclaimer). No architecture, CI/CD, or developer content there.
- **README.md (English) and README.jp.md (Japanese) must stay in sync.
  Whenever you update one, apply the same change to the other.**
- **SPEC.md is the developer document**: feature spec, architecture, dev
  commands, CI/CD, and release procedure. Update it when behavior changes.

## Trademark rules

- The extension is named "Lifecycle Highlighter **for** OpenShift" — keep the
  "for" form; never rename it in a way that implies it is an official Red Hat product.
- Never use Red Hat or OpenShift logos (or look-alikes) in icons or store assets.
  The current icons are original geometric shapes generated by `PIL` — keep it that way.
- Keep the "unofficial / not affiliated with Red Hat" disclaimer in both READMEs,
  the store listing, and `extDescription`.

## Key architectural constraints (do not regress)

- The lifecycle table lives inside the **Shadow DOM** of the `<plcc-table>` Lit
  component. Any DOM lookup must recursively traverse shadow roots.
- Column identification uses each cell's `data-label` / `headers` attributes —
  never rely on cell index or site CSS class names.
- Styles must be injected as `<style data-ocp-lh>` into each shadow root;
  a manifest-declared CSS file cannot reach Shadow DOM.
- `src/lib.js` is pure logic shared by the content script, options page, unit
  tests, and `scripts/check-structure.mjs`. It must stay free of DOM and
  chrome.* dependencies, and keep working both as a browser global (`OCPLH`)
  and as a CommonJS module.
- Cells that don't parse as dates (`N/A`, `GA of 4.22 + 3 Months`) are skipped.
  The GA column is excluded from highlighting.

## Commands

```bash
npm test                                      # unit tests (node:test, no deps)
npm run build                                 # zip into dist/
npm run check:structure                       # live-page validation (playwright)
node scripts/check-structure.mjs --api-only   # API check only (no playwright needed)
```

## Workflow notes

- `_locales/en` and `_locales/ja` must have identical message keys (CI enforces this).
- Release: bump `version` in **both** `manifest.json` and `package.json`, then push
  a matching `v*` tag. The release workflow verifies tag == manifest version.
- `structure-check.yml` runs daily against the live Red Hat page and API; if it
  fails, the page structure likely changed — fix `src/lib.js` detection logic and
  `scripts/check-structure.mjs` expectations together, and verify with a real
  Playwright run before releasing.
